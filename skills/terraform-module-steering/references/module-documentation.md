# Module documentation — make specs, features, and controls referenceable

Documentation is part of **"done"** — for both new and changed modules. A module without a clear
consumer README and a preserved design record is not finished. Two deliverables:

1. A **consumer `README.md`** — how to use the module.
2. A **durable design record** — the steering/change doc preserved *with* the module, so the specs,
   features, and security-control rationale survive for future reference.

**Truth rule:** the README's security section must match what's actually enforced (and verified by
checkov). Never document a control you didn't implement — no aspirational claims.

## 1. Consumer README.md

Adapt this structure to the module:

```markdown
# terraform-module-<provider>-<service>
<one-line purpose>

## Features
- what it deploys, and the security guarantees (hardcoded, non-overridable)

## Security controls enforced
| Control (CIS / provider id) | Enforcement | Overridable? |
|---|---|---|
| <id> | <hardcoded setting> | No — hardcoded |
<!-- consumers see exactly what they get, and that it can't be weakened -->

## Usage
​```hcl
module "<name>" {
  source      = "<registry-or-git-source>"
  name        = "<name>"
  environment = "prod"
}
​```

## Requirements
| Name | Version |
|------|---------|
| terraform | >= x.y |
| <provider> | >= a.b |

## Inputs
<!-- BEGIN_TF_DOCS -->
<!-- generated by terraform-docs, or a hand-written table -->
<!-- END_TF_DOCS -->

## Outputs
<!-- (same) -->

## Examples
- [minimal](examples/minimal)

## Upgrade notes        <!-- existing modules only -->
- vX.Y.Z — <feature / gap closed>; <breaking changes + migration, or "no breaking changes">

## License
```

### Keep Inputs/Outputs in sync with terraform-docs
Prefer generating the Inputs/Outputs tables so they never drift:
```bash
terraform-docs markdown table --output-file README.md --output-mode inject .
```
This fills between the `BEGIN_TF_DOCS`/`END_TF_DOCS` markers. If `terraform-docs` isn't installed (the
workspace check reports it), hand-write the tables from `variables.tf`/`outputs.tf`. A pre-commit
`terraform_docs` hook keeps them fresh. The **Security controls enforced** and **Usage** sections are
always hand-written — terraform-docs won't produce them.

## 2. Durable design record (specs / features / controls for reference)

Preserve the steering doc (greenfield) or change-steering doc (brownfield) with the module so the
rationale outlives the conversation:

- Put it at `<module>/docs/DESIGN.md` (or `docs/steering/<date>-<topic>.md` for a history).
- Keep the **security control mapping** (control → enforcement → justification) there, mirrored by the
  README's "Security controls enforced" table.
- For significant either/or decisions, a short ADR under `docs/adr/NNNN-title.md` is encouraged.
- For a **changed** module, also append a `CHANGELOG.md` entry: the feature/gap closed + the
  recommended semver bump + any migration. (Write the content only — performing the commit/tag is the
  creator's job; this skill does no git.)

## Hard rules
- README + design record are part of "done"; name them in the steering doc's deliverables.
- README controls section must match the verified, enforced controls.
- No git operations — produce the files locally; the creator commits.
