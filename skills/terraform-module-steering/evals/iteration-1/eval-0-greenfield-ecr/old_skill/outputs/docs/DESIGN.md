# Steering Document — `terraform-module-aws-ecr`

> Binding context for building the module. Generated by the `terraform-module-steering` skill,
> greenfield (Path A). Local content only — **no git operations** performed.

## 1. Identity
- **Module name:** `terraform-module-aws-ecr`
- **Provider / service:** AWS / Elastic Container Registry (ECR), private repository.
- **Purpose (one line):** A secure-by-default private ECR repository that satisfies the mandatory AWS
  FSBP ECR controls with a minimal consumer interface.
- **Target environments:** dev / test / staging / prod.

## 2. Reuse decision
- **Wrap or scratch:** **Wrap** `terraform-aws-modules/ecr/aws` pinned **`3.2.0`** (exact pin).
- **Why:** The upstream module already exposes every input needed for the FSBP controls — tag
  mutability, scan-on-push, encryption type + KMS key, and a lifecycle policy — and renders the
  `aws_ecr_repository` + `aws_ecr_lifecycle_policy` resources for us. Wrapping inherits its maintenance
  and lets us hardcode security by passing literals and exposing no override variables. Confirmed the
  interface by downloading the pinned module and reading its real `variables.tf` / `outputs.tf` /
  `main.tf` / `versions.tf` (see BUILD_LOG.md "How the interface was confirmed").
- **Provider + Terraform version floor:** terraform **`>= 1.5.7`**, aws provider **`>= 6.28`** (matches
  upstream v3.2.0's `versions.tf`). Built/verified with Terraform v1.15.6 (latest supported, June 2026)
  and aws provider v6.50.0.

## 3. Consumer interface (minimise inputs)
- **Required inputs:**
  - `name` (string) — the ECR repository name.
  - `environment` (string) — validated to one of `dev` / `test` / `staging` / `prod`.
- **Optional inputs (safe defaults):**
  - `tags = {}` — merged UNDER the module's mandatory tags (consumer cannot drop `Environment` /
    `ManagedBy`).
  - `force_delete = false` — allow `terraform destroy` of a repo that still holds images. Defaults to
    the safe value `false`.
  - `untagged_image_expiry_days = 14` — lifecycle: expire untagged images after N days (the value of
    the FSBP ECR.3 lifecycle policy is tunable; *that* a policy exists is hardcoded).
  - `max_tagged_image_count = 100` — lifecycle: keep at most N tagged images.
- **NOT exposed (hardcoded — the non-overridable guarantee):**
  - `image_tag_mutability = "IMMUTABLE"` (ECR.2)
  - `scan_on_push = true` (ECR.1)
  - encryption: KMS with a module-managed customer-managed CMK (ECR.5) — neither the encryption type
    nor the key is a consumer variable.
  - *that* a lifecycle policy is attached (ECR.3) — only its thresholds are tunable.
  - KMS key rotation (`enable_key_rotation = true`) and deletion window.
- **Outputs:** `repository_name`, `repository_arn`, `repository_url`, `repository_registry_id`,
  `kms_key_arn`, `kms_key_id`, `kms_alias_arn`, `kms_alias_name`.
- **Provider config:** the module does NOT configure the aws provider; the consumer owns region/creds.

## 4. Security controls (researched, proposed, negotiated)
**Benchmarks used:** AWS Foundational Security Best Practices (FSBP), the AWS Security Hub standard, for
ECR. **CIS AWS Foundations Benchmark has no ECR-specific controls** — confirmed by research; the ECR
posture lives in FSBP (and CIS EKS for the cluster side, which is out of scope here). Said plainly per
the skill's "services with little CIS posture" guidance.

| Control (id) | Intent | Enforcement (hardcoded literal) | Decision | Justification |
|---|---|---|---|---|
| **FSBP ECR.1** | Detect vulnerable images | `repository_image_scan_on_push = true` | kept | scan every pushed image |
| **FSBP ECR.2** | Trusted, static tags | `repository_image_tag_mutability = "IMMUTABLE"` | kept | a tag cannot be silently overwritten |
| **FSBP ECR.3** | Avoid stale/unbounded images | `create_lifecycle_policy = true` + a non-empty lifecycle policy JSON | kept | at least one lifecycle policy is always attached |
| **FSBP ECR.5** | Encrypt at rest with a CMK | `repository_encryption_type = "KMS"` + module-managed `aws_kms_key` ARN; `enable_key_rotation = true` | kept | full key control + annual rotation |
| **FSBP ECR.4** | Tag public repos | — | removed (N/A) | this module creates **private** repos only; ECR.4 targets `AWS::ECR::PublicRepository` |
| Account/registry-level scanning (`ENHANCED`) | Org-wide Inspector scanning | — | removed (out of scope) | registry scanning config is account-wide, not per-repo; documented as a recommendation |

_Custom benchmarks: none requested (automated run). A removed control has a recorded reason above._

### Scanner reality (checkov)
- The upstream module configures `image_scanning_configuration`, `encryption_configuration`, and the
  lifecycle policy as nested blocks / a separate `aws_ecr_lifecycle_policy` resource. Under checkov's
  **static directory scan of `examples/minimal`**, checkov follows the downloaded external module and
  evaluates the real `aws_ecr_repository` — so ECR.1/ECR.2/ECR.3-style checks (`CKV_AWS_51` immutable
  tags, `CKV_AWS_136` KMS encryption, `CKV_AWS_163` scan-on-push) are expected to **pass** because the
  literals reach the resource. Any residual finding on an upstream-internal resource we cannot annotate
  is suppressed in `.checkov.yaml` with a documented reason. The KMS key policy document we own carries
  an inline `#checkov:skip` only if a true false-positive appears.

## 5. Conventions
- **Files:** `versions.tf`, `variables.tf`, `main.tf` (wrapper call + lifecycle policy local),
  `kms.tf` (CMK + alias + key policy), `outputs.tf`.
- **Block ordering:** resource → `count`/`for_each` first, then args, then `tags`, then `lifecycle`.
  Variables → description → type → default → validation.
- **Version pinning:** upstream module pinned to exact `3.2.0`; aws provider floor `>= 6.28`.
- **Tagging:** mandatory tags (`Environment`, `ManagedBy = "terraform"`, `Module`) merged OVER consumer
  tags so a consumer cannot drop them.
- **`this`** reserved for the single wrapper instance / single KMS key.

## 6. Verification pipeline (definition of done)
- `terraform fmt -check -recursive`
- `terraform init -backend=false`
- `terraform validate`
- `tflint --init && tflint` (aws ruleset)
- `terraform test` (native, `mock_provider` — no creds; mock `aws_iam_policy_document.json` with valid
  JSON so the upstream repository-policy document doesn't break apply)
- `checkov -d examples/minimal --download-external-modules true --config-file .checkov.yaml`
- **Done = all green, with evidence.**

## 7. Examples, tests & documentation to include
- `examples/minimal/` — `name` + `environment` only; doubles as the checkov fixture.
- `tests/*.tftest.hcl` — assert: KMS rotation on; KMS literal encryption type wired; tag immutability =
  IMMUTABLE; scan-on-push true; lifecycle policy created & non-empty; mandatory tags present & not
  droppable; environment validation rejects bad values; outputs populated.
- `README.md` — consumer docs incl. a **Security controls enforced** table (all hardcoded /
  non-overridable).
- `docs/DESIGN.md` — this steering doc preserved as the durable design record.

## 8. Out of scope / limitations
- Public ECR repositories (`aws_ecrpublic_repository`) and FSBP ECR.4.
- Account/registry-level enhanced scanning, registry replication, pull-through cache, registry policy —
  all account/registry-wide, not per-repository.
- Repository IAM policy is left at the upstream default (account-root read + read/write ARNs); no extra
  cross-account policy is layered by this module.

## 9. Hard rules
- Security controls are **hardcoded literals, not variables**.
- **No git operations** are performed while building from this document.
