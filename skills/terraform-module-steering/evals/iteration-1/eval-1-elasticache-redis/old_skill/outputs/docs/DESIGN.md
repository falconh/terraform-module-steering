## 1. Identity
- **Module name:** `terraform-module-aws-elasticache-redis`
- **Provider / service:** AWS / ElastiCache for Redis (replication group)
- **Purpose:** A secure-by-default Amazon ElastiCache for Redis **replication group** that satisfies the
  mandatory AWS FSBP ElastiCache controls with a minimal interface, a customer-managed KMS key, enforced
  TLS, a write-only AUTH token that never lands in Terraform state, multi-AZ + automatic failover on by
  default, and slow-log delivery to CloudWatch.
- **Target environments:** dev / test / staging / prod.

## 2. Reuse decision
- **Wrap or scratch:** **Wrap** `terraform-aws-modules/elasticache/aws` pinned **`1.11.0`** (latest, released 2026-01-29).
- **Why:** It exposes every input needed for the controls ŌĆö `at_rest_encryption_enabled`, `kms_key_arn`,
  `transit_encryption_enabled`, `transit_encryption_mode`, `multi_az_enabled`, `automatic_failover_enabled`,
  `num_cache_clusters`, `log_delivery_configuration` (incl. self-managed CloudWatch log group creation),
  `snapshot_retention_limit`, `auto_minor_version_upgrade`, custom subnet group ŌĆö so wrapping inherits its
  maintenance and lets us hardcode security by passing literals and exposing no override variable.
- **Confirmed upstream interface (how):** read the v1.11.0 source directly ŌĆö
  `variables.tf`, `outputs.tf`, `main.tf`, `versions.tf` on GitHub at tag `v1.11.0`. Verified that the
  replication-group resource derives `automatic_failover_enabled = var.multi_az_enabled || var.cluster_mode_enabled ? true : var.automatic_failover_enabled`, that it creates its own `aws_cloudwatch_log_group.this[*]` from `log_delivery_configuration` entries whose `destination_type == "cloudwatch-logs"`, and that `auth_token` is a plain (state-persisted) string argument.
- **Version floor:** terraform **`>= 1.11`** (write-only arguments require 1.11; ephemeral input variables require 1.10 ŌĆö we take the stricter 1.11); aws provider **`>= 6.0`** (latest at build time: 6.50.0; upstream module floor is `>= 5.93` but we raise it because the write-only `secret_string_wo` path needs a 6.x provider); random provider `>= 3.0` (transitively required by upstream).

- **Outputs:** `replication_group_id`, `replication_group_arn`, `primary_endpoint_address`,

## 4. Security controls (researched, proposed, negotiated)
**Benchmarks used:** AWS Foundational Security Best Practices (FSBP) ŌĆö the Security Hub ElastiCache
controls. **Note on CIS:** the CIS AWS Foundations Benchmark has **no ElastiCache-specific controls**
(it focuses on IAM, logging, monitoring, networking, storage). FSBP is therefore the governing
service-specific benchmark here; CIS account-wide controls (CloudTrail, KMS rotation) are honored where
they touch our resources (we rotate the CMK). This is stated plainly per the skill's generalisation rule.

| Control (id) | Intent | Enforcement (hardcoded literal) | Decision | Justification |
|---|---|---|---|---|
| FSBP ElastiCache.4 | Encrypt at rest with a CMK | `at_rest_encryption_enabled = true`, `kms_key_arn = aws_kms_key.this.arn` | kept | task requires customer-managed KMS at rest |
| FSBP ElastiCache.5 | Encrypt in transit (TLS) | `transit_encryption_enabled = true`, `transit_encryption_mode = "required"` | kept | task requires TLS enforced (not merely "preferred") |
| FSBP ElastiCache.6 | Redis AUTH enabled | AUTH token stored write-only in Secrets Manager (`secret_string_wo`); transit encryption on | kept (adapted) | task: AUTH must be write-only / never in state ŌĆö see write-only mechanism below |
| FSBP ElastiCache.3 | Automatic failover | `automatic_failover_enabled = true` (also forced by `multi_az_enabled = true` in upstream) | kept | task requires failover ENABLED by default |
| FSBP ElastiCache.1 | Automatic backups | `snapshot_retention_limit = 7` (>= 1) | kept | recoverability; 7 days is a sensible prod default |
| FSBP ElastiCache.2 | Auto minor version upgrades | `auto_minor_version_upgrade = true` | kept | security patching |
| FSBP ElastiCache.7 | No default subnet group | `create_subnet_group = true` from consumer `subnet_ids` | kept | network isolation; custom group required |
| Multi-AZ (HA, beyond FSBP) | High availability across AZs | `multi_az_enabled = true` | added | task requires multi-AZ by default; emits a cost WARNING |
| CMK key rotation (CIS-aligned) | Annual key rotation | `enable_key_rotation = true` on the CMK | added | CIS KMS rotation posture applied to our CMK |
| Secret encrypted with CMK | AUTH secret at rest | `kms_key_id = aws_kms_key.this.arn` on the secret | added | the secret holding AUTH is itself CMK-encrypted |

No controls removed. No custom benchmarks requested.

### Write-only AUTH token mechanism (the central design fork)
**DECISION:** Research + empirical probes (run during the build) established three facts:
1. `aws_elasticache_replication_group` has **no native write-only `auth_token_wo`** argument
   (open provider issue hashicorp/terraform-provider-aws#42239; absent in provider 6.50.0). Its
   `auth_token` is a normal, **state-persisted** string.
2. Terraform **rejects** an `ephemeral` value flowing into `auth_token` ŌĆö proven:
   `Error: Ephemeral values are not valid for "auth_token", because it is not a write-only attribute and must be persisted to state.`
3. The upstream module's `auth_token` input is not declared ephemeral-accepting, so the ephemeral value
   cannot reach it either (`Error: Ephemeral value not allowed`).

Therefore, to honor *"the AUTH token must never be stored in Terraform state"* the token is **not routed
through the upstream module's `auth_token` argument at all**. Instead:
- The wrapper variable `auth_token` is `ephemeral = true` + `sensitive = true` ŌåÆ never serialized to
  state or plan.
- The module persists it **only** to a module-managed **AWS Secrets Manager** secret via the write-only
  `secret_string_wo` argument (paired with `secret_string_wo_version`), which Terraform discards without
  storing. Verified: an ephemeral var flows cleanly into `secret_string_wo` (validate succeeds).
- The secret is CMK-encrypted. Consumers/apps read the live AUTH token from that secret ARN at runtime;
  the operator applies/rotates it on the cluster out-of-band (documented), keeping TF state secret-free.

This is the only mechanism that is simultaneously (a) write-only / state-free and (b) compatible with the
proven upstream module today. It is documented honestly in the README as the AUTH posture.

### Cost WARNING for multi-AZ (the second fork)
**DECISION:** Multi-AZ + automatic failover roughly doubles node-hours (a replica per primary) and adds
cross-AZ data-transfer cost. The task requires the module to *emit a WARNING at plan/apply* so consumers
see it, AND the pipeline must keep `terraform test` green.

Empirical findings from build-time probes:
- A **`check` block** with a deliberately-false `assert` surfaces a **non-fatal WARNING** at every
  `terraform plan` / `terraform apply` (exactly the desired UX ŌĆö the operator sees the cost note, the run
  still succeeds). This is the right mechanism for the consumer-facing warning.
- **Gotcha (this is the fork the task flagged):** `terraform test` **promotes a failed `check` assertion
  into a test FAILURE** (`0 passed, 1 failed`). So a naked `check` block breaks the test pipeline.
- **Resolution (verified):** in each affected `run` block, declare
  `expect_failures = [check.multi_az_cost_warning]`. This tells `terraform test` the warning is
  *expected*, so the run **passes** while the warning is still emitted to real consumers at plan/apply.
  Probe result: `terraform test` ŌåÆ `1 passed, 0 failed` with the warning present.

Final mechanism: `check "multi_az_cost_warning"` (assert intentionally false, with the cost message in
`error_message`) lives in `main.tf`; every test `run` that applies the module lists it in
`expect_failures`. See BUILD_LOG.md for the exact probe transcripts.

## 8. Out of scope / limitations
- Account-/org-wide settings (CloudTrail, Config, Security Hub enablement).
- RBAC user groups (Redis 6+) ŌĆö deferred; AUTH token is the requested mechanism.
- Native `auth_token_wo` on the cluster ŌĆö not available in the provider yet (issue #42239); revisit when shipped to route AUTH straight to the cluster write-only.
- Global datastore / cross-region replication, data tiering, custom parameter tuning.
- The actual application of the AUTH token onto the running cluster is an out-of-band/runtime step (documented), because the cluster argument is not write-only.

## 9. Hard rules
- Security controls are **hardcoded literals, not variables**.
- The AUTH token is **never** written to state, plan, outputs, or a `.tfvars`.
- **No git operations** are performed while building from this document.
